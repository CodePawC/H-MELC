# 三、编码体系分层
建议：
# 四层身份体系
---
# 第一层：系统内部唯一ID（UUID）
用于：
# 数据库内部真实主键
永不变化。
---

```

```

不要暴露给用户。
---

用于：
# 人工识别

```

```

这是：

---

# 四、一机一码设计（核心）
注意：
# 一机一码 ≠ 一个二维码
真正的是：
# “一个设备唯一数字身份”
二维码只是入口。
---
# 五、设备编码规则（推荐）
建议：
# 可解析编码
---
## 推荐结构：
```
[医院]-[对象]-[分类]-[年份]-[序号]
```

---
## 示例：
```
WLH-EQ-MRI-2026-000001
```

含义：
| 部分     | 含义      |
|--------|---------|
| WLH    | 五莲县人民医院 |
| EQ     | 医学装备    |
| MRI    | MRI分类   |
| 2026   | 年份      |
| 000001 | 流水号     |


---
# 六、为什么必须可解析
未来：
# AI和统计会大量依赖编码结构。

快速分析：
  * 哪年采购最多
  * 哪类设备维修率高
  * 哪院区设备老化严重

# 八、二维码体系设计（非常关键）
二维码不能只是：
# 一个URL
否则未来会出问题。
---
# 九、推荐二维码内容结构
建议：
# Token化二维码
---
## 二维码实际内容：
```
{ "asset_code": "WLH-EQ-CT-2026-000001", "token": "xxxxxx", "version": 1}
```

---
# 十、为什么不用纯URL
因为：
| 风险    | 问题      |
|-------|---------|
| URL变化 | 二维码失效   |
| 外网暴露  | 安全问题    |
| 多系统迁移 | 历史二维码失效 |
| 离线扫码  | 无法支持    |


---
# 十一、二维码访问机制
扫码后：
# 不直接进入页面
而是：
```
扫码→ 网关解析→ Token校验→ 用户身份识别→ 权限判断→ 跳转对应功能
```

---
# 十二、同一设备不同角色看到不同页面
这是必须的。
---
## 护士扫码
看到：
  * 报修
  * 使用说明
  * 点检记录
  * 借用申请


---
## 工程师扫码
看到：
  * 维修记录
  * 电路图
  * PM
  * 配件库存
  * AI维修助手


---
## 院领导扫码
看到：
  * ROI
  * 使用率
  * 故障率


---
# 十三、二维码必须支持离线机制（重要）
医院：
  * 电梯井
  * 地下机房
  * 放疗机房


经常无网络。
---
# 十四、建议：
