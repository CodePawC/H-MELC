# 统一报修中心设计

## 一、定位与边界

统一报修中心是医院医学装备运营平台的系统级报修服务能力，不等同于普通“报修工单”页面。它负责接收电脑端、移动端、设备二维码、微信公众号、企业微信、飞书、钉钉、系统内 AI 聊天框、语音、图片、设备科人工代录和系统预警等入口，把多渠道原始消息统一转换、识别、确认、匹配设备，并在条件满足后生成标准化 `repair_order` 工单。

新的模块结构为：

```text
运维中心
├─ 统一报修中心
│  ├─ 报修工作台
│  ├─ 新建报修
│  ├─ AI报修助手
│  ├─ 多渠道接入
│  ├─ 待确认报修
│  ├─ 待派工工单
│  ├─ 维修处理中
│  ├─ 待验收工单
│  ├─ 已完成工单
│  └─ 报修规则配置
```

`repair_order` 只是统一报修中心输出的标准工单结果。聊天消息、语音、图片、二维码入口、机器人入口和预警自动报修必须先进入统一报修消息中枢，不能绕过识别确认机制直接落成正式工单。

## 二、接入入口

统一报修中心一期和二期接入范围如下：

| 入口 | 说明 | 处理方式 |
|---|---|---|
| 电脑端后台报修 | 设备科、护士站、管理端人工填报 | 直接生成消息草稿，确认设备后转工单 |
| 手机端登录报修 | H5 / App 登录后报修 | 绑定用户和科室，支持附件 |
| 设备二维码扫码报修 | 扫设备身份二维码后点击一键报修 | 二维码仍归属资产中心，报修中心只消费设备身份 |
| 微信公众号报修 | 临床人员通过公众号发文字、语音、图片 | 网关回调后标准化为统一消息 |
| 企业微信报修 | 企业微信应用或机器人 | 回调、用户映射、附件下载 |
| 飞书机器人报修 | 飞书机器人消息 | 统一消息解析与交互确认 |
| 钉钉机器人报修 | 钉钉机器人消息 | 统一消息解析与交互确认 |
| 系统内 AI 聊天框 | 平台右下角 AI 报修助手 | 聊天式识别、追问、确认 |
| 语音报修 | 系统内、微信、企业微信、飞书、钉钉语音 | ASR 转文字，原始语音和转写文本均留存 |
| 图片报修 | 故障照片、屏幕报警、铭牌照片 | 保存附件，AI 提取设备和故障线索 |
| 设备科人工代录 | 电话、口头、纸质记录转录 | 标记 `MANUAL_ENTRY` 来源 |
| 系统预警自动生成 | IoT、PM、计量、风险预警转报修 | 标记自动来源，按规则确认或升级 |

## 三、统一处理流程

```text
用户发送文字 / 语音 / 图片 / 视频 / 预警事件
↓
统一报修消息接入层保存原始消息
↓
语音转文字 / 附件入库 / 渠道用户映射
↓
AI识别报修意图
↓
抽取科室、位置、设备、故障现象、紧急程度、是否影响诊疗
↓
系统匹配设备档案
↓
高置信度：展示识别结果让用户确认
中置信度：展示疑似设备列表供用户选择
低置信度：进入待确认报修池
↓
用户或设备科确认
↓
生成标准 repair_order
↓
进入设备科审核、派工、维修、验收、关闭、维修档案沉淀
```

急救类、生命支持类设备即使识别不完整，也必须优先推送设备科值班人员，并在工作台、机器人和值班消息中以危急优先级呈现。

## 四、AI报修助手

AI报修助手支持系统内聊天框和外部机器人交互。用户可输入或说出故障情况，AI 负责抽取候选信息和追问缺失信息。

示例：

```text
用户：ICU 5床监护仪血压打不上去，一直报警。

AI识别：
科室：ICU
位置：5床
设备类型：病人监护仪
故障现象：无创血压测量异常、报警
故障分类：参数异常 / NIBP模块异常
紧急程度：较急
是否影响诊疗：是
是否疑似急救/生命支持类设备：是

AI追问：
请问是否影响当前患者使用？
设备屏幕是否有报警代码？
是否方便上传一张故障照片？
是否需要设备科立即处理？
```

AI只能生成识别结果、候选设备、追问问题和工单草稿。正式工单生成需要用户确认、规则允许或设备科人工确认。低置信度、设备不明确、跨科室权限不足、急救设备信息缺失时不得盲目生成工单。

## 五、识别确认机制

| 匹配级别 | 建议阈值 | 处理方式 |
|---|---:|---|
| 高置信度 | `matched_confidence >= 0.85` | 展示识别结果，用户点击“确认并生成工单”后转工单 |
| 中置信度 | `0.60 <= matched_confidence < 0.85` | 展示疑似设备列表，用户或设备科选择后转工单 |
| 低置信度 | `< 0.60` | 进入待确认报修池，需设备科确认 |
| 急救/生命支持设备 | 任意置信度 | 优先推送值班人员，可先建危急任务，补齐设备后转正式工单 |

疑似设备列表应展示设备名称、型号、科室、位置、资产编号、状态、匹配度和近 30 天维修次数。常用操作为：确认并生成工单、重新选择设备、补充描述、转人工确认。

## 六、语音与图片报修

语音报修包括系统内聊天框语音、微信语音、企业微信语音、飞书语音、钉钉语音。处理流程为：

```text
接收语音
↓
保存原始文件
↓
语音转文字
↓
保存 transcribed_text
↓
AI识别故障内容
↓
抽取关键信息
↓
匹配设备档案
↓
确认后生成工单
```

语音识别后的文本必须保留在 `unified_repair_message.transcribed_text` 中，语音文件保留在 `voice_file_url` 或附件记录中，用于追溯和质控。

图片报修用于识别设备铭牌、屏幕报警、故障现场和附件状态。图片不能单独作为正式工单结论，必须与文字、语音转写或人工补充说明合并后确认。

## 七、多渠道接入配置

多渠道接入页面支持配置：

| 配置项 | 说明 |
|---|---|
| 渠道名称 | 院内展示名称 |
| 渠道类型 | WECHAT_MP / WEWORK / FEISHU / DINGTALK / SMS / INTERNAL_MESSAGE / APP_H5 / DEVICE_QR |
| 启用状态 | 是否接收该渠道消息 |
| 机器人名称 | 外部机器人或应用名称 |
| Webhook地址 | 平台主动推送地址 |
| Token / Secret | 回调校验与签名密钥 |
| 消息回调地址 | 外部平台回调到平台的地址 |
| 支持消息类型 | TEXT / VOICE / IMAGE / VIDEO / FILE / CARD |
| 默认处理规则 | 自动识别、人工确认、仅通知等 |
| 绑定科室范围 | 限定哪些科室可用 |
| 绑定用户范围 | 限定用户或角色 |
| 是否允许自动生成工单 | 受总规则和渠道规则共同约束 |
| 是否需要人工确认 | 高风险或低置信度时强制确认 |

外部机器人还应支持报修进度查询、补充照片、设备科待处理提醒、工程师派工提醒、科室验收提醒、逾期提醒和急救设备即时推送。

## 八、系统内 AI 聊天框

平台右下角增加“AI报修助手”悬浮入口，支持：

| 能力 | 说明 |
|---|---|
| 文字报修 | 输入故障描述并识别 |
| 语音报修 | 录音转文字并保留原始消息 |
| 图片上传 | 上传故障照片、报警代码、铭牌 |
| 故障问答 | 查询常见故障与处理建议 |
| 报修进度查询 | 查询本人或本科室历史报修 |
| 自助排查 | 根据设备和故障推荐排查步骤 |
| 设备档案推荐 | 结合维修历史、PM、计量、知识库推荐建议 |
| 生成报修工单 | 确认后转 `repair_order` |

示例：

```text
用户：内镜中心 290主机图像闪。
AI：我识别到您可能要报修一台电子内镜主机。请确认以下信息：
设备：电子内镜主机 CV-290
科室：内镜中心
故障：图像闪烁
紧急程度：较急
是否影响当前检查工作？
按钮：确认报修 / 更换设备 / 补充照片 / 取消
```

## 九、待确认报修池

无法准确匹配设备的聊天报修、语音报修、微信报修、机器人报修和自动预警进入待确认报修池。

字段包括：消息来源、发送人、发送科室、发送时间、原始内容、语音转文字内容、图片附件、AI识别设备、AI识别科室、AI识别故障、匹配置信度、疑似设备列表、处理状态和操作。

操作包括：确认设备并生成工单、补充信息、指派人员核实、忽略、退回用户补充信息。忽略、退回和人工确认必须写入日志。

## 十、设备二维码关系

设备二维码仍属于资产中心和设备档案的一部分，不属于报修模块。正确关系为：

```text
设备建档
↓
生成设备身份二维码
↓
扫码进入设备移动端详情页
↓
用户点击一键报修
↓
进入统一报修中心
↓
生成统一报修消息
↓
确认后生成报修工单
```

不得把二维码设计为孤立“报修二维码”。二维码代表设备身份，报修只是扫码后的一个入口能力。

## 十一、工单闭环

无论用户通过哪个入口报修，最终都生成统一 `repair_order` 并进入同一套闭环：

```text
报修提交
↓
AI识别 / 人工确认
↓
生成工单
↓
设备科审核
↓
派工
↓
工程师接单
↓
维修处理
↓
科室验收
↓
工单关闭
↓
写入设备维修档案
```

工单关闭后，需要沉淀设备维修档案、维修知识库、故障统计、风险预警和同设备短期多次报修记录。涉及急救、生命支持、高价值或疑似不良事件的工单，应联动质量安全和统一任务引擎。

## 十二、规则配置

报修规则配置页面至少支持：

| 规则 | 说明 |
|---|---|
| 是否允许AI自动生成工单 | 总开关；关闭时只能生成待确认消息 |
| 不同渠道是否需要人工确认 | 渠道粒度覆盖总规则 |
| 急救设备是否自动升级紧急程度 | 默认为启用 |
| 夜间报修是否推送值班人员 | 支持值班时间段和人员组 |
| 超时未派工提醒规则 | 按优先级设置提醒阈值 |
| 超时未验收提醒规则 | 科室和设备科双向提醒 |
| 高价值设备报修通知设备科主任 | 支持资产原值阈值 |
| 同一设备短期多次报修风险预警 | 触发故障频发风险 |
| 生命支持类设备提示备用设备调配 | 与应急调配联动 |
| 临床用户是否可查看维修进度 | 按角色和科室范围控制 |

| 表 | 作用 |
|---|---|
| `repair.repair_order` | 标准报修工单结果表 |
| `repair.repair_order_log` | 工单流转日志 |
| `repair.repair_attachment` | 报修附件 |

## 十四、验收要点

统一报修中心验收不只验证能否提交表单，而应验证：

- 多渠道入口都进入 `unified_repair_message`。
- AI识别结果需要确认，低置信度进入待确认池。
- 语音原文件和转写文本均可追溯。
- 图片、语音、视频附件与消息和工单双向关联。
- 高风险和急救设备报修优先通知值班人员。
- 设备二维码保持资产身份入口，不被改造成单一报修码。
- 所有入口最终进入统一 `repair_order` 闭环。
- 机器人和系统内聊天框都支持进度查询。
- 规则配置可影响不同渠道和不同设备类型的处理策略。
