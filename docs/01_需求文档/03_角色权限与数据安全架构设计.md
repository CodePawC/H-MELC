# 权限体系与数据安全架构设计

# 《权限体系与数据安全架构设计》
这个平台涉及供应商、财务、审计、设备科、临床科室、AI知识库和医院集成平台，权限必须从一开始设计好。
核心原则：
# “最小权限 + 数据隔离 + 全程留痕 + AI不越权”
---
## 一、权限模型
建议采用：
# RBAC + 数据范围权限 + 操作权限 + 流程权限
不要只做简单角色。
| 权限类型   | 示例                  |
|--------|---------------------|
| 角色权限   | 工程师、护士长、财务、供应商      |
| 数据范围权限 | 只能看本科室、本人供应商、本院区    |
| 操作权限   | 新增、修改、删除、审核、导出      |
| 流程权限   | 派单、付款审核、竞价确认、不良事件审核 |
| 字段权限   | 财务金额、供应商报价、合同附件可见性  |
| AI权限   | 能否使用AI、能否查看AI分析依据   |


---
## 二、核心角色设计
| 角色     | 权限重点             |
|--------|------------------|
| 使用科室人员 | 扫码报修、查看本科室设备     |
| 科室负责人  | 查看本科室设备、维修、借用、效益 |
| 工程师    | 接单、维修、PM、巡检、知识库  |
| 设备科管理员 | 台账、派单、流程、证照、统计   |
| 设备科主任  | 审核、风险处置、报表、供应商评价 |
| 财务科    | 发票、付款、账龄、付款计划    |
| 审计科    | 竞价、合同、付款链路、审计报告  |
| 院领导    | 驾驶舱、运营分析、风险预警    |
| 供应商    | 仅查看本单位项目、发票、付款状态 |
| 系统管理员  | 用户、权限、配置、接口、日志   |


---
## 三、供应商门户隔离
供应商门户必须独立隔离。
建议：
  * 独立域名或路径
  * 独立认证策略
  * 独立菜单
  * 独立数据范围
  * 严格接口鉴权
  * 不允许访问医院内部管理页面
  * 不允许查看其他供应商数据
  * 不允许查看患者信息
  * 不允许查看非公开设备信息


供应商能看到：
| 可见内容    | 说明          |
|---------|-------------|
| 自己的维修项目 | 已报名/已中标/已完成 |
| 自己的耗材订单 | 订单、验收、发票    |
| 自己的发票   | 上传、审核、付款状态  |
| 自己的付款进度 | 已付、未付、部分付款  |
| 待补材料    | 注册证、随货同行、资质 |
| 公告类项目   | 可参与报价的公开项目  |


供应商不能看到：
  * 其他供应商报价
  * 全院设备明细
  * 医院财务总盘
  * 患者相关数据
  * 内部审批意见原文
  * 非公开采购计划


---
## 四、数据范围权限
建议每次查询都带数据范围过滤。
| 用户   | 数据范围                |
|------|---------------------|
| 科室人员 | 本科室设备、自己提交的报修       |
| 护士长  | 本科室全部设备与报修          |
| 工程师  | 自己工单 + 授权设备范围       |
| 设备科  | 全院设备                |
| 财务   | 发票、付款、合同金额          |
| 审计   | 合同、竞价、付款链路          |
| 院领导  | 汇总数据为主，可下钻授权        |
| 供应商  | supplier_id = 当前供应商 |


---
## 五、字段级权限
尤其要保护：
| 字段     | 权限控制         |
|--------|--------------|
| 合同金额   | 财务、审计、设备科主任  |
| 供应商报价  | 竞价结束前严格隔离    |
| 付款计划   | 财务、院领导       |
| 欠款总额   | 财务、院领导、设备科主任 |
| 不良事件详情 | 设备科、质控、授权领导  |
| AI分析依据 | 按业务权限继承      |
| 接口密钥   | 仅系统管理员       |


---
## 六、AI权限边界
AI必须遵守用户权限。
原则：
# 用户看不到的数据，AI也不能回答。
例如：
供应商问：
> “医院一共欠所有供应商多少钱？”
AI必须拒绝。
科室问：
> “其他科室设备使用率怎么样？”
只能在授权后回答。
工程师问维修案例，可以只返回脱敏知识。

## 七、审计日志设计
所有关键动作必须记录：
| 动作      | 是否必须留痕 |
|---------|--------|
| 登录      | 是      |
| 查看财务数据  | 是      |
| 下载合同/发票 | 是      |
| 修改设备台账  | 是      |
| 修改付款状态  | 是      |
| 竞价报价    | 是      |
| 审批通过/驳回 | 是      |
| AI生成报告  | 是      |
| 删除/作废记录 | 是      |
| 权限变更    | 是      |


审计日志字段建议：
| 字段          | 说明   |
|-------------|------|
| user_id     | 操作人  |
| role_id     | 当前角色 |
| action      | 操作   |
| object_type | 对象类型 |
| object_id   | 对象ID |
| before_data | 变更前  |
| after_data  | 变更后  |
| ip_address  | IP   |
| user_agent  | 终端   |
| created_at  | 时间   |


---
## 八、安全认证建议
| 场景    | 建议                   |
|-------|----------------------|
| 内部用户  | 医院统一身份认证/LDAP/OAuth2 |
| 供应商   | 独立账号 + 手机/邮箱验证       |
| 高风险操作 | 二次验证                 |
| API调用 | JWT + 签名 + 过期时间      |
| 移动端   | Token刷新机制            |
| PDA   | 设备绑定                 |
| 第三方接口 | 白名单 + 签名             |


---
## 九、文件安全
平台会保存大量敏感文件：
  * 发票
  * 合同
  * 注册证
  * 随货同行
  * 维修报告
  * 不良事件资料
  * 设备说明书


建议：
  * 文件统一进入文件中心
  * MinIO/对象存储
  * 文件权限继承业务对象
  * 文件下载留痕
  * 文件防篡改校验
  * 敏感文件水印
  * 外部供应商上传文件需病毒扫描
  * 重要文件版本化


---
## 十、接口安全
与医院集成平台对接时建议：
  * 所有接口通过API网关
  * 不允许数据库直连
  * 服务间使用内部Token
  * 接口访问留痕
  * 接口限流
  * 关键接口幂等
  * 敏感接口加签
  * 数据同步失败可重试
