# 医用装备全生命周期智能管理平台 - PRD产品需求文档

可以。根据你的描述，这个平台不应只是“设备台账系统”，而应定位为：
# 医用装备全生命周期智能管理平台
核心目标是：  
以**一机一码** 为入口，以**设备全生命周期状态流转** 为主线，以**合规监管、维修闭环、效能分析、知识库沉淀、AI辅助决策** 为能力中枢，形成覆盖医疗设备、医用器械、耗材、气体、证照、维修、采购、竞价、效益分析的综合管理平台。
---
## 一、我对系统的初步归纳
### 1\. 核心主线
每一台设备从进入医院开始，系统都应形成完整链路：
**采购论证 → 招采/合同 → 到货验收 → 建账赋码 → 安装培训 → 使用运行 → 巡检保养 → 维修报修 → 计量检测 → 质量安全 → 效益分析 → 调拨借用 → 更新报废 → 档案归档**
所有状态都要可追溯、可查询、可统计、可审计。
---
## 二、一期建议优先建设模块
我建议一期不要一上来全部铺开，而是先做“主干闭环”。
### 一期核心模块
| 模块       | 建设目标                       |
|----------|----------------------------|
| 设备主数据/台账 | 一机一码、分类、型号、厂家、供应商、科室、状态、档案 |
| 扫码报修     | 支持图文、语音、视频、AI初步判断          |
| 工程师维修工作台 | 接单、派单、维修记录、费用判断、AI辅助分析     |
| 预防性维护    | PM计划、执行记录、逾期提醒             |
| 计量检测     | 强检/非强检、周期、证书、到期预警          |
| 巡检管理     | 设备巡检、急救生命支持类设备完好率          |
| 应急调配     | 借用、归还、跨科室调配、使用频率分析         |
| 证照合规     | 放射、辐射、压力容器、特种设备、大型设备配置证    |
| 设备效能分析   | 使用率、收入、成本、ROI、闲置预警         |
| AI知识库    | 维修案例、制度流程、故障经验、报告生成        |

## 三、关键业务闭环设计
### 1\. 报修闭环
建议流程：
**科室扫码 → 自动识别设备 → 提交故障描述/语音/图片/视频 → AI预判故障类型 → 自动分级 → 工程师接单/强制派单 → 维修处理 → 是否需费用/第三方/返厂 → 科室确认 → AI生成维修报告 → 进入知识库 → 必要时触发不良事件流程**
这里有一个重要判断点：
> 不是所有维修都只是维修。  
>  系统必须识别：是否存在医疗器械不良事件、是否影响患者安全、是否需要暂停使用、是否需要上报。
---
### 2\. 第三方维修/竞价闭环
建议流程：
**工程师发起外修/费用申请 → 科室/设备科审核 → 对外平台发布 → 第三方注册/资质上传 → 报价/接单 → 竞价 → 审计科/财务科审核 → 确认服务商 → 维修执行 → 验收评价 → 费用归档**
这个模块涉及外部访问，必须单独设计：
  * 第三方账号隔离
  * 只允许看到公开项目
  * 上传营业执照、授权书、维修资质
  * 报价留痕
  * 审计可追溯
  * 防止私下修改报价
  * 竞价过程可导出审计报告


---
### 3\. AI分析闭环
AI不应只是聊天框，而应嵌入每个场景：
| 场景    | AI能力                  |
|-------|-----------------------|
| 报修    | 故障初判、风险分级、建议是否停用      |
| 工程师维修 | 故障树分析、维修步骤建议、备件建议     |
| 报告生成  | 自动生成维修报告、季度工作量报告      |
| 不良事件  | 判断是否疑似不良事件，生成初步调查框架   |
| 应急调配  | 分析借用频次，建议调拨或采购        |
| 效益分析  | 分析使用率、收入、成本、ROI       |
| 知识库   | 将琐碎工作沉淀为标准案例、FAQ、制度依据 |

## 五、技术架构初步判断
你给出的技术栈是合理的：
  * 后端：FastAPI + Python 3.11
  * 数据库：PostgreSQL 16
  * ORM：SQLAlchemy 2.x + Alembic
  * 缓存/队列：Redis
  * 前端：Vite + React + TypeScript
  * 部署：Docker
  * CI：GitHub Actions
  * 文件处理：Excel、Word、附件上传


建议补充：
| 能力    | 建议                                      |
|-------|-----------------------------------------|
| 异步任务  | Celery / RQ / Dramatiq                  |
| 文件存储  | MinIO 或院内对象存储                           |
| 搜索能力  | PostgreSQL full-text，后期可接 Elasticsearch |

| 外部平台  | 与内网主系统物理/逻辑隔离，采用API网关                   |

## 六、我认为必须尽早明确的关键问题
### 1\. “设备”和“器械/耗材”是否统一建模？
建议采用统一模型：
**医学装备资产对象 Asset**  
下面再分：
  * 固定资产设备
  * 低值设备
  * 生命支持类设备
  * 急救设备
  * 放射设备
  * 计量设备
  * 压力容器/特种设备
  * 医用耗材
  * 高值耗材
  * 医用气体相关设备


这样后期扩展更稳。
---
### 2\. 一机一码的编码规则
建议不要只用资产编号。应建立平台自己的唯一编码：
**医院代码 + 资产类别 + 年份 + 序号 + 校验码**
例如：
`WLH-EQ-2026-000001`
二维码扫码后进入设备详情页，并根据用户角色展示不同功能。
---
### 3\. AI是否允许直接给结论？
建议AI只做：
  * 辅助判断
  * 风险提示
  * 报告草拟
  * 知识检索
  * 决策建议


不应直接替代人工确认。所有关键节点必须有人确认。
