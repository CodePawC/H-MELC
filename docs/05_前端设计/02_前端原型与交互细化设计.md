# 《前端原型与页面交互细化设计 V1.0》

## 一、设计原则

平台前端要围绕四个核心体验：

|原则|说明|
|---|---|
|一线人员少填|科室扫码报修要快|
|工程师少找|工单、资料、AI建议集中展示|
|管理者少算|统计、风险、付款自动汇总|
|供应商少问|发票、付款、材料状态透明|

# 五、统一报修中心

## 交互目标

统一报修中心不是单一表单页，而是多渠道报修消息中枢。临床科室通过扫码、移动端、微信、企业微信、飞书、钉钉、语音、图片或系统内 AI 聊天框发起报修后，平台应在 30 秒内完成消息入池、AI 初识别和下一步确认提示；正式工单生成前必须经过用户确认、规则放行或设备科人工确认。

## PC 报修工作台布局

```text
顶部统计卡片
├─ 今日报修
├─ AI识别报修
├─ 微信报修
├─ 飞书报修
├─ 待确认消息
├─ 待派工工单
├─ 维修中工单
├─ 急救设备故障
└─ 超时未处理

左侧渠道筛选
中间消息/工单列表
右侧详情抽屉
```

## 多渠道接入页面

支持配置微信公众号、企业微信、飞书机器人、钉钉机器人、短信入口、内部系统消息、App / H5 移动端和设备二维码扫码入口。每个渠道配置渠道名称、渠道类型、启用状态、机器人名称、Webhook 地址、Token / Secret、消息回调地址、支持消息类型、默认处理规则、绑定科室范围、绑定用户范围、是否允许自动生成工单、是否需要人工确认。

## 报修规则配置页面

页面采用 Ant Design Pro 表单和分组卡片：

- AI自动生成工单总开关。
- 不同渠道是否需要人工确认。
- 急救设备是否自动升级紧急程度。
- 夜间报修是否推送值班人员。
- 超时未派工提醒规则。
- 超时未验收提醒规则。
- 高价值设备报修是否通知设备科主任。
- 同一设备短期多次报修是否触发风险预警。
- 生命支持类设备故障是否提示备用设备调配。
- 是否允许临床用户查看维修进度。

## 设备二维码边界

设备二维码仍属于资产中心和设备档案。移动端扫码后先进入设备详情页，再通过“一键报修”进入统一报修中心。页面不得把二维码描述为单纯“报修二维码”。

# 七、AI报修助手与AI维修助手交互

## 入口

- 平台右下角 AI报修助手悬浮入口
- 统一报修中心 / AI报修助手
- 工单详情页右侧
- AI中心
- 知识库问答

## 交互内容

临床用户可输入：

```
ICU 5床监护仪血压打不上去，一直报警。
```

AI返回：

- 科室：ICU
- 位置：5床
- 设备类型：病人监护仪
- 故障现象：无创血压测量异常、报警
- 故障分类：参数异常 / NIBP模块异常
- 紧急程度：较急
- 是否影响诊疗：是
- 是否疑似急救/生命支持类设备：是
- 疑似设备列表及匹配度
- 追问：是否影响当前患者使用、是否有报警代码、是否上传照片、是否需要立即处理

工程师可输入：

```
监护仪NIBP模块压力异常，反复报错，如何排查？
```

AI返回：

- 可能原因
- 排查步骤
- 相似案例
- 所需备件
- 风险提示
- 引用来源

## 重要提示

页面必须显示：

```
AI结果仅供参考。报修识别结果需要用户或设备科确认，最终维修结论由工程师确认。
```

### PC管理端一期（与 `frontend-admin` 对齐）

- **文档列表**：各知识库菜单入口共用分页检索列表；条目进入详情 **`/knowledge/documents/{document_id}`**。
