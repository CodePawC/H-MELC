# PC管理端一期实现对照

本文档描述仓库内 **`frontend-admin`（React + TypeScript + Vite）** 与 **`docs/06_接口设计/01_API接口设计.md`** 的**已实现**对接关系，供验收、联调与迭代排期使用。  
**不替代**《前端页面与菜单结构设计》中的远期菜单规划；未实现能力在工程中仍以占位页呈现。

---

## 一、约定

| 项 | 说明 |
|----|------|
| API 前缀 | `/api/v1` |
| 响应信封 | `{ code, message, data }`（见接口总册说明） |
| 鉴权 | Bearer JWT；角色与接口的 RBAC 以 `backend/app/modules/auth/rbac.py` 及各路由 `dependencies` 为准 |
| 错误呈现 | 403/404/409/503 等由页面展示接口返回的 `detail` 或业务 `message` |
| 配置 | 管理端 `VITE_API_BASE_URL`，默认对接本机后端（见 `frontend-admin/.env.example`） |

---

## 二、动态路由（不在侧栏叶子表中展开）

| 路径 | 页面 | 主要接口 |
|------|------|----------|
| `/lifecycle/assets/:assetId` | 设备详情（多标签） | `GET /assets/{id}`；标准分类页签展示 H-UMDG 医疗器械分类目录关联与影响提醒 |
| `/repair/tickets/:repairOrderId` | 报修工单详情（统一报修中心生成后的标准工单详情，多标签 + 运维动作） | `GET /repairs/{id}` 及同章各 `POST` 动作 |
| `/maintenance/tickets/:repairOrderId` | 旧路径兼容 | 自动兼容到同一详情页 |
| `/knowledge/documents/:documentId` | 知识中心文档详情 | `GET /knowledge/documents/{id}` |

## 四、统一报修中心（PC）行为摘要

对齐 **`docs/06_接口设计/01`** 第二章（统一报修中心 §二·1～11，以及标准工单 §二·12～19；外修、返厂相关 POST 仍见第三章）。

| 能力 | 说明 |
|------|------|
| 报修工作台 | 路由 `/repair/center`；展示今日报修、AI识别报修、多渠道报修、待确认、待派工、维修中、急救设备故障、超时未处理等统计；后续对接 `GET /repair-center/messages` |
| 新建报修 | 路由 `/repair/new`；用于设备科代录或后台报修，先生成统一报修消息，再确认转工单 |
| AI报修助手 | 路由 `/repair/ai-assistant`；聊天式识别文字、语音、图片，展示候选设备和追问问题；后续对接 `/repair-center/ai-sessions` |
| 多渠道接入 | 路由 `/repair/channels`；配置微信公众号、企业微信、飞书、钉钉、短信、内部消息、App/H5、设备二维码入口 |
| 待确认报修池 | 路由 `/repair/pending-confirmations`；低置信度或设备不明确的消息由设备科确认后转 `repair_order` |
| 报修规则配置 | 路由 `/repair/rules`；配置 AI 自动生成、渠道确认、急救设备升级、夜间值班、SLA 超时、高价值设备通知、重复报修风险等规则 |
| 工单列表 | `GET /repairs`，支持文档所列查询参数：`order_status`、`asset_id`、`department_id`、`assigned_engineer_id`、`priority`、`date_from`、`date_to`、分页 `page`/`page_size` |
| 工程师派单列表 | 同 `GET /repairs`；路由 `/repair/dispatch`，默认 `order_status=PENDING_DISPATCH`；URL 中 `order_status=*` 表示不限状态 |
| 地址栏筛选 | 上述条件可体现在 URL 查询串中并与表单双向同步（便于自设备详情深链：`/maintenance/tickets?asset_id={uuid}` 或 `/maintenance/dispatch?asset_id={uuid}`） |
| 工单详情 | **工单概览、附件、维修过程、维修报告**；数据来源 `GET /repairs/{id}` 聚合 |
| 运维动作 | 独立标签：抢单、派单、过程记录、完成维修、科室确认、外修/返厂标记；分别对应同路径下各 `POST`；工程师 ID 可预填 **`GET /auth/me`** 的 `id`（JWT subject） |
| 二维码边界 | 设备二维码仍在资产中心生成和维护；统一报修中心只把扫码后的“一键报修”作为入口来源 |

设备详情页的 **维修记录** 表中：工单号链至工单详情；并提供 **在本设备维度打开报修列表** 的链接（带 `asset_id`）。
