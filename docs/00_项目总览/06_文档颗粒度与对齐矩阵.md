# 文档颗粒度与对齐矩阵

本文用于统一本项目文档的“写到什么深度、放在哪个文件、变更时同步哪里”。后续新增需求、接口、页面、数据表、测试用例时，优先按本文对齐，避免 SRS、PRD、接口、前端实现、测试验收之间粒度漂移。

---

## 一、文档层级

| 层级 | 目标 | 写作颗粒度 | 主文档 |
|---|---|---|---|
| L0 项目总览 | 说明为什么建设、建设路线、交付边界、文档治理规则 | 战略目标、阶段边界、文档规则，不写具体字段和接口细节 | `docs/00_项目总览/` |
| L1 需求范围 | 说明业务要做什么、一期/二期/远期分别做什么 | 到业务模块、角色、状态、核心流程；不写数据库字段清单 | `docs/01_需求文档/` |
| L2 架构与数据 | 说明系统如何拆分、数据如何建模、流程如何运行 | 到服务边界、核心实体、状态机、关键表、索引策略 | `docs/02_系统架构/`、`docs/03_数据库设计/`、`docs/04_AI设计/` |
| L3 接口与前端 | 说明工程如何对接、页面如何落地 | 到 API 路径、请求/响应、菜单路由、页面状态、已实现/占位状态 | `docs/05_前端设计/`、`docs/06_接口设计/` |
| L4 测试与上线 | 说明如何验证、运行、部署、验收 | 到测试场景、契约用例、探针、部署步骤、上线检查项 | `docs/07_部署运维/`、`docs/08_测试验收/`、`docs/09_实施上线/` |

---

## 二、颗粒度规则

| 内容类型 | 推荐颗粒度 | 不应写入 |
|---|---|---|
| SRS | 平台目标、用户角色、系统范围、一期/二期/三期边界、非功能要求 | 具体接口字段、React 组件名、SQL 细节 |
| PRD | 业务场景、用户旅程、主流程、异常流程、页面能力描述 | 后端实现路径、Alembic 版本号、测试命令 |
| 权限与安全 | 角色、数据范围、操作权限、流程权限、安全原则 | 某个页面按钮的样式、接口返回示例 |
| 业务流程/状态机 | 生命周期节点、状态流转、触发条件、AI 介入点 | 表字段逐项说明 |
| 架构设计 | 模块边界、服务职责、集成方式、关键技术约束 | 每个页面的展示文案 |
| 数据库设计 | schema、核心表、关键字段、约束、索引、枚举 | 需求背景长段落 |
| AI 设计 | AI 任务边界、输入输出、人工复核、知识库/RAG 策略 | 前端菜单逐项实现状态 |
| 前端菜单/交互 | 菜单树、页面行为、路由、筛选、跳转、空态/错误态 | 后端表结构与迁移脚本 |
| PC 实现对照 | 当前 `frontend-admin` 已接通、占位、回退演示数据、源码索引 | 远期愿景性需求 |
| API 契约 | `/api/v1` 路径、鉴权、请求参数、响应信封、错误码、503 就绪约束 | 页面视觉布局说明 |
| 测试验收 | 验收范围、用例编号、预期结果、契约测试约束、上线判定 | 产品愿景、架构推演 |
| 开发实施 | 本地运行、依赖、测试命令、端口、文档站、迭代顺序 | 业务规则原始讨论 |

## 四、变更同步规则

| 变更类型 | 必须同步 | 建议同步 |
|---|---|---|
| 新增业务能力 | SRS/PRD 或对应需求模块、里程碑范围、验收范围 | 前端菜单树、数据模型、AI 设计 |
| 新增或修改 API | `docs/06_接口设计/01_API接口设计.md`、`backend/tests/` 中 contract/doc 用例 | `docs/05_前端设计/03_PC管理端一期实现对照.md`、开发指南 |
| 前端页面从占位变为已接通 | `docs/05_前端设计/03_PC管理端一期实现对照.md` | `docs/08_测试验收/01_验收标准与测试用例设计.md` |
| 新增数据库表/字段/枚举 | `docs/03_数据库设计/` 对应表结构、字段、索引或枚举文档 | API 契约、测试数据说明 |
| 新增 AI 能力 | `docs/04_AI设计/`、`docs/06_接口设计/01_API接口设计.md` | 验收用例、知识库资料索引 |
| 调整部署/健康检查/环境变量 | `docs/07_部署运维/`、`docs/09_实施上线/02_开发实施指南.md` | README、AGENTS |
| 调整一期/二期边界 | `docs/01_需求文档/06_开发里程碑与一期建设范围.md` | PC 实现对照、测试验收范围、交付清单 |

## 五、章节模板

新增业务模块文档建议保持以下最小结构：

```md
# 模块名称

## 一、定位与边界

## 二、角色与权限

## 三、业务流程

## 四、状态与数据口径

## 五、接口与页面映射

## 六、验收要点

## 七、二期扩展
```

新增接口章节建议保持以下最小结构：

```md
## N. 能力名称

路径：`METHOD /api/v1/...`

鉴权：角色 / 数据范围

请求参数：

响应 `data`：

错误码与就绪约束：

关联页面：

契约测试：
```

## 六、当前整理优先级

1. 优先把 **一期已接通能力** 的链路补齐：设备台账、一机一码、报修维修、PM、计量合规、知识库、系统用户与角色。
2. 对 **已接通但验收用例仍粗略** 的模块补专项用例：PM、计量合规、供应商协同、财务付款、运营中心。
3. 对 **远期规划** 保留在 SRS/PRD/里程碑，不下沉到接口字段和页面源码索引。
4. 对 **工程实现变化** 优先更新 `docs/05_前端设计/03_PC管理端一期实现对照.md` 与 `docs/06_接口设计/01_API接口设计.md`，再补测试验收。
