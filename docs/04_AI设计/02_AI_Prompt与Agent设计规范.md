# AI Prompt体系与Agent设计规范 V1.0

# 《AI Prompt体系与Agent设计规范 V1.0》
## 一、核心原则
AI在本平台中必须遵守四条底线：
  1. **只辅助，不替代责任人决策**
  2. **只基于授权数据回答**
  3. **所有重要结论必须可追溯**
  4. **所有AI结果必须可人工审核、修改、驳回**


---
# 二、AI输出统一格式
所有AI分析结果建议统一为 JSON + 文本摘要双结构。
```
{ "summary": "简要结论", "risk_level": "LOW/MEDIUM/HIGH/CRITICAL", "confidence": 0.86, "basis": ["引用的设备记录", "维修历史", "知识库文档"], "recommendations": [], "need_human_review": true}
```

---
# 三、Agent分类
| Agent      | 主要场景           |
|------------|----------------|
| 报修分诊Agent  | 报修后判断故障类型、风险等级 |
| 维修诊断Agent  | 工程师排查故障        |
| 维修报告Agent  | 自动生成维修报告       |
| 不良事件Agent  | 判断是否疑似不良事件     |
| 票据识别Agent  | 发票、随货同行、注册证OCR |
| 对账Agent    | 发票、合同、验收、付款匹配  |
| 付款优先级Agent | 分析付款轻重缓急       |
| 效益分析Agent  | ROI、使用率、调拨建议   |
| 知识库问答Agent | 制度、维修案例、厂家资料问答 |


---
# 四、报修分诊Agent
## 输入
  * 设备信息
  * 科室
  * 报修描述
  * 图片/视频/语音转写
  * 历史维修记录
  * 设备风险标签


## 输出
```
{ "fault_category": "NIBP_MODULE", "priority": "P2", "suggest_stop_use": true, "possible_incident": false, "recommended_engineer_tags": ["监护仪", "迈瑞", "NIBP"], "triage_reason": "该设备属于生命支持相关设备，故障影响监测功能。"}
```

---
# 五、维修诊断Agent
## 输出内容
  * 可能故障原因
  * 排查步骤
  * 所需工具
  * 备件建议
  * 相似案例
  * 风险提示
  * 是否建议外修
  * 是否建议停用


关键限制：
> AI不得直接判定设备可继续临床使用，只能建议工程师确认。
---
# 六、维修报告Agent
维修完成后生成：
```
设备基本信息故障现象故障原因分析维修过程更换配件功能测试结果安全风险判断是否建议纳入不良事件观察科室确认意见
```

报告必须标记：
  * AI生成
  * 工程师审核
  * 科室确认
  * 版本号


---
# 七、不良事件Agent
用于判断是否“疑似不良事件”。
判断依据包括：
  * 是否涉及患者伤害或潜在伤害
  * 是否重复发生
  * 是否存在设计缺陷可能
  * 是否为生命支持/高风险设备
  * 是否造成治疗中断
  * 是否需停用隔离


输出：
```
{ "possible_adverse_event": true, "risk_level": "HIGH", "suggested_action": "建议发起医疗器械不良事件初步调查流程", "reason": "同型号设备短期内出现多起相似故障，且涉及生命体征监测功能。"}
```

---
# 八、票据识别Agent
识别对象：
  * 发票
  * 随货同行
  * 注册证
  * 合格证
  * 维修报价单
  * 合同扫描件


输出字段：
```
{ "supplier_name": "", "invoice_number": "", "invoice_code": "", "invoice_date": "", "total_amount": 0, "tax_amount": 0, "items": [ { "name": "", "model": "", "quantity": 0, "unit_price": 0, "amount": 0 } ], "confidence": 0.92, "need_manual_check": true}
```

---
# 九、对账Agent
对账逻辑：
```
发票→ 供应商→ 合同→ 订单→ 到货验收→ 入库→ 应付款→ 付款记录
```

AI可以提示：
  * 金额不一致
  * 发票重复
  * 合同号缺失
  * 供应商名称不一致
  * 已付款但未结清
  * 可能重复付款


---
# 十、付款优先级Agent
分析维度：
| 维度   | 说明               |
|------|------------------|
| 临床影响 | 是否影响急救、ICU、手术、放疗 |
| 欠款账龄 | 超期天数             |
| 供应风险 | 是否可能停供/拒修        |
| 金额大小 | 金额压力             |
| 历史合作 | 履约质量             |
| 合规风险 | 是否存在审计风险         |
| 领导关注 | 是否重点项目           |


输出：
```
{ "supplier_id": "uuid", "priority_level": "HIGH", "suggested_payment_amount": 50000, "reason": "该供应商承担DSA急修服务，欠款账龄超过180天，存在后续拒修风险。"}
```

---
# 十一、效益分析Agent
输入：
  * 设备原值
  * 折旧
  * 使用次数
  * 收入
  * 维修成本
  * 停机时长
  * 科室需求
  * 借用频率


输出：
  * 使用率评价
  * ROI趋势
  * 是否闲置
  * 是否建议调拨
  * 是否建议新增采购
  * 是否建议报废更新


---
# 十二、知识库问答Agent
必须遵守：
# “先检索，后回答”
回答时必须返回：
  * 来源文档
  * 来源段落
  * 可信度
  * 是否缺少证据


如果知识库没有资料，应回答：
> 当前知识库未检索到可靠依据，建议补充厂家手册或制度文件。

| 字段                   | 说明               |
|----------------------|------------------|
| template_code        | REPAIR_TRIAGE_V1 |

| version              | v1.0             |
| system_prompt        | 系统提示词            |
| user_prompt_template | 用户提示词模板          |

| status               | ACTIVE/INACTIVE  |
| created_by           | 创建人              |


---
# 十四、AI审核机制
AI结果状态：
```
PENDING_REVIEW→ ACCEPTED→ MODIFIED→ REJECTED
```

所有被采用的AI结论，都必须记录：
  * 原始AI输出
  * 人工修改后内容
  * 审核人
  * 审核时间
  * 应用到哪个业务对象
