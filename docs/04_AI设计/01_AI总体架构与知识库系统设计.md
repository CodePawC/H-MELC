# AI总体架构与知识库系统设计

# 《AI总体架构与知识库系统设计》
# AI Architecture & Knowledge Engine
这一部分是平台的“智能中枢”。
你的系统中，AI不应是一个聊天窗口，而应成为：
# 嵌入每个业务流程的智能决策引擎
---
## 一、AI总体定位
AI主要承担六类任务：
| 类型   | 作用                     |
|------|------------------------|
| 智能识别 | 发票、随货同行、注册证、维修图片、视频、语音 |
| 智能分析 | 故障原因、付款优先级、设备效益、供应商风险  |
| 智能生成 | 维修报告、季度报告、效益报告、不良事件初稿  |
| 智能预警 | 高故障率、计量逾期、证照到期、欠款风险    |
| 智能问答 | 制度、流程、设备资料、维修经验查询      |
| 智能推荐 | 工程师派单、设备调拨、采购建议、付款排序   |


---
## 二、推荐AI架构
建议采用：
```
业务系统 ↓AI服务网关 ↓AI Agent编排层 ↓RAG知识库 / 结构化数据库 / 文件库 / 多模态模型 ↓结果审核与业务回写
```

AI不能直接改业务数据，必须经过人工确认或规则校验。

## 四、向量库建议
你现在使用 PostgreSQL 16，建议一期直接采用：

  * 部署简单

  * 后期可平滑迁移到 Milvus / Qdrant

| 表                   | 作用     |
|---------------------|--------|
| knowledge_document  | 文档主表   |
| knowledge_chunk     | 文档切片   |
| knowledge_embedding | 向量     |
| ai_conversation     | AI会话   |
| ai_analysis_result  | AI分析结果 |
| ai_feedback         | 人工反馈   |

## 五、AI在各模块中的嵌入点
### 1\. 统一报修中心AI
| 环节   | AI动作        |
|------|-------------|
| 多渠道接入 | 识别电脑端、移动端、二维码、微信、企业微信、飞书、钉钉、系统内聊天框、人工代录和系统预警的报修意图 |
| 语音报修 | ASR转文字，保留原始语音与转写文本 |
| 图片报修 | 识别故障照片、报警代码、铭牌或现场线索 |
| 聊天式报修 | 抽取科室、位置、设备、故障现象、紧急程度，并追问缺失信息 |
| 设备匹配 | 根据设备档案、科室、位置、名称、历史维修匹配候选设备 |
| 结果确认 | 高置信度让用户确认，中置信度展示疑似设备列表，低置信度进入待确认池 |
| 急救保障 | 急救/生命支持类设备信息不完整时仍优先通知值班人员 |
| 派单前 | 推荐工程师和响应优先级 |
| 维修中 | 提供故障树、维修手册、相似案例 |
| 维修后 | 自动生成维修报告草稿 |
| 关闭前 | 判断是否疑似不良事件 |

AI报修助手是统一报修中心的交互入口之一，不是独立报修孤岛。它只能生成识别结果、候选设备、追问问题和工单草稿；正式 `repair_order` 生成必须经过用户确认、规则放行或设备科人工确认。低置信度、设备不明确、跨科室权限不足、急救设备信息缺失时，不得盲目生成工单。


---
### 2\. 工程师AI助手
工程师可以输入：
> “监护仪NIBP模块反复报错，压力释放异常，如何判断？”
AI返回：
  * 相似维修案例
  * 可能故障原因
  * 排查步骤
  * 所需工具
  * 备件建议
  * 是否需停用
  * 是否可能构成不良事件

### 3\. 供应商门户AI
| 场景   | AI功能           |
|------|----------------|
| 发票上传 | OCR识别发票号、金额、税额 |
| 随货同行 | 提取批号、注册证、数量    |
| 对账   | 自动匹配合同、订单、验收单  |
| 付款分析 | 计算欠款、账龄、付款比例   |
| 风险排序 | 识别应优先付款供应商     |


---
### 4\. 设备效益AI
AI自动分析：
  * 使用率是否偏低
  * 是否适合调拨
  * 是否应新增采购
  * 是否重复配置
  * 是否维修成本过高
  * 是否达到更新年限
  * ROI是否异常


---
### 5\. 不良事件AI
AI辅助生成：
  * 事件经过
  * 风险分析
  * 初步原因
  * 涉及设备信息
  * 厂家信息
  * 整改措施
  * FMEA分析表
  * 上报建议


但必须人工确认后才能进入正式上报流程。

## 七、AI安全边界
必须明确：
# AI只建议，不直接审批
AI不能直接完成：
  * 报废审批
  * 付款审批
  * 不良事件上报
  * 供应商中标确认
  * 设备停用最终决定
  * 财务付款确认


AI可以做：
  * 提醒
  * 草拟
  * 分析
  * 推荐
  * 预警


最终决策由对应责任人确认。
---
## 八、AI结果可信机制
建议每次AI输出都保存：
| 字段                  | 说明     |
|---------------------|--------|
| prompt_version      | 提示词版本  |
| model_name          | 模型名称   |
| input_hash          | 输入摘要   |
| retrieved_docs      | 引用知识   |
| confidence_score    | 置信度    |
| human_review_status | 人工审核状态 |
| final_decision      | 最终处理结果 |


这对医疗行业非常重要。

| 供应商沟通 | 履约记录   |

| PM巡检  | 维护经验   |
| 设备效益  | 管理建议   |
