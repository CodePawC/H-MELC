# 《工作流引擎详细设计 V1.0》
## 一、模块定位
工作流引擎是平台的“流程中枢”。
它不只处理审批，而是统一管理：
  * 外修审批
  * 竞价审核
  * 发票审核
  * 付款审核
  * 设备报废
  * 型号新增审核
  * 不良事件调查
  * CAPA整改
  * 供应商资质审核
  * 计量逾期整改
  * 证照到期整改


核心原则：
# 流程可配置，业务不写死
---
## 二、总体架构
```
业务模块 ↓ 发起流程工作流引擎 ↓ 创建流程实例节点任务 ↓ 分配处理人用户审批/系统自动处理/AI辅助 ↓ 节点流转业务模块 ↓ 接收流程结果并更新状态
```

---
## 三、核心对象
| 对象                  | 说明     |
|---------------------|--------|
| workflow_definition | 流程定义   |
| workflow_node       | 流程节点   |
| workflow_transition | 节点流转规则 |
| workflow_instance   | 流程实例   |
| workflow_task       | 待办任务   |
| workflow_action_log | 审批操作日志 |


---
## 四、流程定义表
```
CREATE TABLE workflow.workflow_definition ( id UUID PRIMARY KEY, workflow_code VARCHAR(64) UNIQUE NOT NULL, workflow_name VARCHAR(255) NOT NULL, business_type VARCHAR(64) NOT NULL, version INTEGER DEFAULT 1, status VARCHAR(64) DEFAULT 'ACTIVE', description TEXT, created_by UUID, created_at TIMESTAMP DEFAULT now(), updated_at TIMESTAMP DEFAULT now());
```

示例：
| workflow_code             | 含义      |
|---------------------------|---------|
| REPAIR_OUTSOURCE_APPROVAL | 外修审批    |
| PAYMENT_APPROVAL          | 付款审批    |
| SUPPLIER_BID_REVIEW       | 竞价审核    |
| MODEL_MASTER_REVIEW       | 型号主数据审核 |
| ADVERSE_EVENT_PROCESS     | 不良事件流程  |
| SCRAP_APPROVAL            | 报废审批    |


---
## 五、流程节点表
```
CREATE TABLE workflow.workflow_node ( id UUID PRIMARY KEY, workflow_definition_id UUID NOT NULL REFERENCES workflow.workflow_definition(id), node_code VARCHAR(64) NOT NULL, node_name VARCHAR(255) NOT NULL, node_type VARCHAR(64) NOT NULL, assignee_type VARCHAR(64), assignee_value VARCHAR(255), is_start BOOLEAN DEFAULT FALSE, is_end BOOLEAN DEFAULT FALSE, config JSONB, created_at TIMESTAMP DEFAULT now());
```

## 八、流转规则表
```
CREATE TABLE workflow.workflow_transition ( id UUID PRIMARY KEY, workflow_definition_id UUID NOT NULL REFERENCES workflow.workflow_definition(id), from_node_id UUID NOT NULL, to_node_id UUID NOT NULL, action_code VARCHAR(64), action_name VARCHAR(128), condition_expression TEXT, sort_order INTEGER DEFAULT 0, created_at TIMESTAMP DEFAULT now());
```

示例：
| action_code | 含义 |
|-------------|----|
| APPROVE     | 同意 |
| REJECT      | 驳回 |
| RETURN      | 退回 |
| CANCEL      | 撤销 |
| TRANSFER    | 转办 |

## 九、流程实例表
```
CREATE TABLE workflow.workflow_instance ( id UUID PRIMARY KEY, workflow_definition_id UUID NOT NULL REFERENCES workflow.workflow_definition(id), business_type VARCHAR(64) NOT NULL, business_id UUID NOT NULL, title VARCHAR(255), current_node_id UUID, status VARCHAR(64) DEFAULT 'RUNNING', started_by UUID, started_at TIMESTAMP DEFAULT now(), ended_at TIMESTAMP, variables JSONB);
```

---
## 十、任务表
```
CREATE TABLE workflow.workflow_task ( id UUID PRIMARY KEY, workflow_instance_id UUID NOT NULL REFERENCES workflow.workflow_instance(id), node_id UUID NOT NULL, task_title VARCHAR(255), assignee_id UUID, assignee_role VARCHAR(64), task_status VARCHAR(64) DEFAULT 'PENDING', due_at TIMESTAMP, completed_at TIMESTAMP, created_at TIMESTAMP DEFAULT now());
```

任务状态：
| 状态          | 说明  |
|-------------|-----|
| PENDING     | 待处理 |
| COMPLETED   | 已完成 |
| REJECTED    | 已驳回 |
| TRANSFERRED | 已转办 |
| CANCELLED   | 已取消 |
| EXPIRED     | 已超期 |

## 十一、审批日志表
```
CREATE TABLE workflow.workflow_action_log ( id UUID PRIMARY KEY, workflow_instance_id UUID NOT NULL, task_id UUID, action_code VARCHAR(64), action_name VARCHAR(128), operator_id UUID, operator_name VARCHAR(128), comment TEXT, before_node_id UUID, after_node_id UUID, created_at TIMESTAMP DEFAULT now(), metadata JSONB);
```

## 十二、典型流程一：外修审批
```
工程师发起外修→ 设备科主任审核→ 金额判断 ├─ ≤3000：设备科确认 └─ >3000：财务审核→ 审计备案→ 流程结束
```

AI介入：
  * 自动生成故障摘要
  * 判断是否必须外修
  * 推荐供应商
  * 评估历史维修成本


---
## 十三、典型流程二：付款审批
```
财务发起付款计划→ 设备科确认业务真实性→ 财务负责人审核→ 院领导审批→ 登记付款→ 供应商门户更新进度
```

条件分支：
| 条件       | 流程        |
|----------|-----------|
| 金额 ≤ 1万元 | 财务负责人审核   |
| 1万-10万元  | 分管领导审批    |
| >10万元    | 院长审批/集体决策 |


---
## 十四、典型流程三：型号新增审核
```
设备入库发现型号不存在→ 申请临时型号→ MDM管理员审核 ├─ 通过：发布正式型号 ├─ 驳回：退回修改 └─ 合并：关联已有型号→ 通知业务系统更新
```

---
## 十五、典型流程四：不良事件流程
```
AI提示疑似事件→ 设备科确认→ 创建调查→ FMEA评分→ CAPA制定→ 责任科室整改→ 效果验证→ 关闭归档
```

高风险规则：
```
RPN > 240→ 自动要求设备科主任确认是否停用
```

## 十六、SLA机制
每个节点可配置时限：
```
{ "sla_hours": 24, "remind_before_hours": 4, "escalate_after_hours": 8}
```

SLA动作：
  * 到期提醒
  * 超期提醒
  * 自动升级
  * 生成超期记录
  * 纳入绩效统计

|-----------|--------|
| 系统消息      | 所有待办   |
| 企业微信/微信   | 工单、审批  |
| 短信        | 高风险事件  |
| 邮件        | 供应商、财务 |

## 十九、权限控制
工作流必须校验：
  * 当前用户是否为任务处理人
  * 是否拥有审批权限
  * 是否能查看业务对象
  * 是否能查看附件
  * 是否能退回/转办
  * 是否能终止流程


---
## 二十、AI在工作流中的边界
AI可以：
  * 生成摘要
  * 提供建议
  * 判断风险
  * 推荐流程分支
  * 草拟审批意见


AI不能：
  * 自动同意付款
  * 自动确认中标
  * 自动停用设备
  * 自动关闭不良事件
  * 自动代替负责人审批


---
## 二十一、流程版本管理
流程定义必须支持版本。
规则：
  * 已运行流程继续使用旧版本
  * 新发起流程使用最新启用版本
  * 旧版本不可删除，只能停用
  * 修改流程必须生成新版本
