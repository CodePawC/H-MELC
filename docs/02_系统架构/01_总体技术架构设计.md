# Overall Technical Architecture
这一部分开始把前面的业务构想落到真正可开发、可部署、可迭代的技术架构上。
---
# 一、总体架构定位
建议采用：
# “模块化单体优先，微服务化预留”
一期不要直接拆成大量微服务，否则开发、部署、调试、数据一致性都会变复杂。
推荐策略：
```
第一阶段：模块化单体第二阶段：核心模块服务化第三阶段：平台化微服务架构
```

# 七、Redis使用场景
Redis 不只是缓存，建议用于：
| 场景     | 用途            |
|--------|---------------|
| 登录会话   | Token黑名单、刷新状态 |
| 验证码    | 供应商登录、二次验证    |
| 热点字典   | 科室、设备分类、状态字典  |
| 工单通知   | 实时消息队列        |
| AI任务状态 | 任务进度缓存        |
| 限流     | 防止供应商接口刷请求    |
| 分布式锁   | 防止重复接单、重复付款分配 |

# 九、AI服务架构
AI建议独立成内部服务：
```
ai-service/├─ OCR Agent├─ Repair Agent├─ Invoice Agent├─ Knowledge RAG Agent├─ ROI Agent├─ Incident Agent└─ Report Agent
```

AI服务不能直接修改业务表，只能写入：
```
ai_analysis_result
```

业务人员确认后再回写业务结果。

# 十、集成平台对接原则
你的医院集成平台应作为统一数据入口。
原则：
  * 不直连 HIS、LIS、PACS 数据库
  * 通过集成平台 API 获取
  * 主数据从 MDM 获取
  * 收入、检查人次、设备使用量从数据平台同步
  * 所有接口调用留痕
  * 接口失败可重试
  * 建立同步任务状态表

# 十三、部署架构
一期建议：
```
Docker Compose├─ backend├─ frontend├─ postgres├─ redis├─ minio└─ nginx

```
